Vous vous souvenez de [git-moar](/blog/Projects/git-moar) ? Ce petit outil que j'avais construit parce que fixer `git log` me faisait saigner les yeux ? Et ben... je me suis emballé. Vraiment *beaucoup* emballé ^^'

Ce qui a commencé par "vas-y, je mets quelques graphs sur des données git" est devenu un monorepo complet avec un renderer React, un site marketing en TanStack Start, des templates TSX custom, des worklogs générés par IA, et assez de métriques de qualité pour faire pleurer votre chef de projet de joie (ou d'horreur, ça dépend de votre historique de commits).

Je vous présente **git-snitch**. Ce que git-moar est devenu en grandissant.

## Qu'est-ce qui a changé ?

git-moar, ça marchait ! Il faisait le taf ! Des gens (bon, 3 personnes, mais quand même !) l'utilisaient !

Mais le code... c'était moi qui apprenais en live avec l'IA, donc c'était tenu ensemble par des espoirs, des rêves, et une quantité inquiétante de `// TODO: refactor later` que le "later" n'est jamais arrivé.

git-snitch, c'est la même idée — transformer votre historique git en quelque chose de lisible — mais reconstruit de zéro comme un vrai monorepo :

* **`@git-snitch/core`** — tout le calcul de données git, l'analyse, l'anonymisation
* **`@git-snitch/renderer`** — React + Recharts + TanStack Router, génère du HTML autonome
* **`@git-snitch/ui`** — composants shadcn/ui partagés
* **`@git-snitch/cli`** — l'interface en ligne de commande

Oui, j'ai fait un monorepo complet pour un outil qui génère un seul fichier HTML. Oui, je vois l'ironie. Non, je le regrette pas (enfin presque).

## Trois commandes, c'est tout

```bash
# Analyser un dépôt unique
git-snitch repo

# Scanner un dossier et découvrir tous les dépôts git
git-snitch scan ~/projects

# Générer un worklog IA à partir de données exportées
git-snitch worklog report.json
```

Ça, c'est tout le CLI. `repo` pour un projet, `scan` pour plusieurs, `worklog` quand vous avez besoin d'un résumé narratif au lieu de graphs. Simple.

## Les trucs biens

### Des métriques de qualité qui veulent vraiment dire quelque chose

git-snitch calcule de vrais signaux de qualité à partir de votre historique de commits :

* **Bus Factor** — combien de gens doivent disparaître avant que votre projet meure (spoiler : pour la plupart des side projects, c'est 1, et ce 1 c'est vous)
* **Code Churn** — lignes ajoutées vs supprimées dans le temps, avec la répartition par contributeur
* **Health Score** — un seul chiffre pour vous dire si votre repo est en pleine forme ou sous assistance respiratoire
* **Coefficient de Gini** — ouais, le truc d'économie, appliqué à l'inégalité des contributions. Parce que visiblement il me fallait rendre l'analyse git *académique*

### L'anonymisation qui respecte votre vie privée

Ça, j'en suis plutôt fier. Vous devez partager un rapport mais vous voulez pas que tout internet connaisse votre arborescence de fichiers et les emails de l'équipe ?

```bash
git-snitch repo --anon
```

Un seul flag. Les noms deviennent des pseudonymes, les emails sont remplacés, les chemins de fichiers sont hashés en FNV-1a, les hashes de commits sont brouillés, les messages sont remplacés par des labels de classification. Vous pouvez même utiliser `--obfuscate-key` pour du hash déterministe, comme ça les rapports anonymisés restent comparables d'une exécution à l'autre.

Ou alors vous cherry-pickez ce que vous voulez cacher :

```bash
git-snitch repo --hide-emails --hide-urls --hide-messages
```

### Les worklogs générés par IA

C'était la feature "oh, je peux faire ça ?". Donnez les données git-snitch à un harness IA et récupérez un work log narratif :

```bash
git-snitch repo --wl-harness opencode --wl-skill work-log
```

Compatible avec [opencode](https://opencode.ai), [pi](https://github.com/nicepkg/pi), et [Codex](https://github.com/openai/codex) comme harness. Quatre templates de skills : `repo-log`, `work-log`, `changelog`, `devlog`. Choisissez votre saveur.

### Des templates custom

Vous aimez pas mon magnifique (enfin, disons fonctionnel) rapport par défaut ? Passez un fichier TSX :

```bash
git-snitch repo --template ./my-template.tsx
```

Les routes manquantes se rabattent sur le renderer intégré, donc vous pouvez surcharger juste les parties qui vous intéressent sans tout réimplémenter.

### Le scan multi-repo

La commande `scan` découvre récursivement les dépôts git et produit un rapport comparatif. Vous voulez voir tout votre workspace d'un coup d'œil ?

```bash
git-snitch scan ~/projects --period 3m --max-depth 3
```

Analyse cross-projet des contributeurs, tendances d'activité comparatives, le tout cuit.

## La stack technique

Je sais ce que vous pensez. "C'est un outil qui génère du HTML, pourquoi il a besoin de tout ça ?" Et vous avez raison ! Mais je voulais aussi apprendre des trucs, et la combo de l'IA qui me rendait plus rapide et de la SEP qui me ralentissait voulait dire que je pouvais me permettre de partir un peu en couilles.

* **TypeScript** partout, mode strict, pas de `any` autorisé
* **pnpm + Turborepo** monorepo
* **React 19 + Recharts** pour le renderer de rapports
* **TanStack Router** pour la navigation dans les rapports
* **TanStack Start** pour le site marketing
* **shadcn/ui** partagé via un package workspace
* **Vite** pour builder le renderer autonome
* **Commander** pour le CLI
* **Zod** pour la validation
* **Vitest** pour les tests

Chaque rapport est un **fichier HTML unique et autonome**. Pas de serveur. Pas de base de données. Pas d'hébergement nécessaire. Ouvrez-le depuis votre système de fichiers, envoyez-le par mail, collez-le sur un drive partagé. Les graphs sont intégrés, les styles sont inline. Ça marche et c'est tout.

## L'enrichissement GitHub

Si le CLI `gh` est installé et que vous avez un remote GitHub, git-snitch récupère automatiquement les stars, forks, licence et topics pour les inclure dans le rapport. Vous voulez pas de ça ? `--no-github` et on en parle plus. Simple.

## Configuration

Créez `.git-snitch/config.json` dans votre repo ou dossier de scan :

```json
{
  "report": {
    "outputPath": "./reports/output.html",
    "theme": "dark"
  },
  "repo": {
    "branches": ["main", "develop"]
  },
  "anon": {
    "hideEmails": true,
    "hideUrls": true
  }
}
```

Les flags CLI surchargent les valeurs de config. Des défauts sensés partout. Pas besoin de fichier de config pour démarrer.

## C'est quoi ce nom ?

"git-snitch" parce que ça balance vos habitudes git. Qui commit à 3h du mat' ? Qui fait des merge commits de 2000 lignes ? Qui a pas touché un fichier depuis 6 mois ? git-snitch va vous le dire. Il va le dire à tout le monde.

(La feature d'anonymisation existe précisément à cause de ça ^^')
