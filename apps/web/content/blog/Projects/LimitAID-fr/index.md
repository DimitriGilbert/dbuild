Alors voilà le truc avec l'utilisation intensive de l'IA en tant que dev — vous traversez les rate limits comme si y'avait pas de lendemain. Et chaque fournisseur a sa propre façon de vous montrer ce qu'il vous reste.

Codex ? Faut fouiller dans leur API backend. OpenRouter ? Un endpoint différent, un format différent. Z.ai ? Encore une autre API avec encore une autre structure de réponse. Et aucun d'eux a une simple commande "hé, il me reste combien de temps ?".

J'en ai eu marre de jongler entre les onglets du navigateur et les docs d'API, alors j'ai fait ce que tout dev raisonnable ferait : j'ai écrit un script bash.

Je vous présente **LimitAID**. Parce que visiblement même mes outils IA ont besoin de premiers soins quand j'en ai fini avec eux.

## Ça fait quoi ?

LimitAID, c'est un CLI bash qui interroge vos fournisseurs d'IA et vous affiche les quotas, soldes et utilisation — tout depuis un seul endroit, dans votre terminal, avec des jolies couleurs et des barres de progression.

```bash
limitaid all
```

C'est tout. Une commande. Trois fournisseurs. Tout ce que vous avez besoin de savoir.

## Les fournisseurs

### Codex / OpenAI (+ Spark)

```bash
limitaid codex
```

Affiche votre type de plan, l'utilisation de la fenêtre principale (en général 5h), la fenêtre secondaire (hebdomadaire), le solde de crédits, et les limites Spark si vous en avez. Il découvre tout seul votre token depuis `~/.codex/auth.json`, donc même pas besoin de configurer quoi que ce soit.

### OpenRouter

```bash
limitaid openrouter
```

Solde de la clé, crédits restants, utilisation daily/hebdomadaire/mensuelle, statut free-tier. Supporte plusieurs clés — parce que bien sûr vous avez une clé pro et une clé perso (mens pas, on le fait tous).

### Z.ai

```bash
limitaid zai
```

Niveau de plan, utilisation de la fenêtre de coding de 5h, limites de tokens hebdomadaires, et — ça c'est le truc cool — une répartition par-MCP de votre utilisation des outils. Vous voulez savoir combien vous avez claqué sur search-prime vs web-reader vs zread ? Maintenant vous le pouvez. Découvre automatiquement votre clé depuis `~/.config/opencode/opencode.json`.

### Tous d'un coup

```bash
limitaid all
```

Lance les trois fournisseurs à la suite. Boum. Vue d'ensemble complète.

## Le mode loop

Ça, je l'utilise *tout le temps*. Quand je suis en pleine session de coding IA et que je dois garder un œil sur mes limites sans lâcher ce que je fais :

```bash
limitaid all --loop 60
```

Efface le terminal et redessine toutes les 60 secondes (minimum 30s, je suis pas un monstre). Un dashboard pour votre quota IA, direct dans votre terminal. C'est comme avoir une jauge de carburant pour votre vitesse de codage.

## La sortie JSON

Pour quand vous voulez piper des trucs dans d'autres trucs (on aime tous piper des trucs dans d'autres trucs) :

```bash
limitaid codex --json
limitaid all --json
```

Renvoie les payloads bruts des réponses des fournisseurs. Balancez ça dans `jq`, pipez dans un script, construisez votre propre dashboard, faites ce qui vous chante.

## Configuration

LimitAID supporte trois façons de lui fournir les clés API, par ordre de priorité :

**1. Override CLI** (priorité la plus haute) :

```bash
limitaid zai --provider-config 'zai::your-api-key::personal'
```

**2. Fichier de config** (`~/.config/limitaid/keys.conf`) :

```ini
openrouter_work=/home/user/.secrets/openrouter_work
openrouter_personal=/home/user/.secrets/openrouter_personal
zai_personal=/home/user/.secrets/zai_personal
```

**3. Auto-découverte** (fallback) :

Codex lit depuis `~/.codex/auth.json`, Z.ai lit depuis `~/.config/opencode/opencode.json`. OpenRouter fait pas d'auto-découverte parce que... ben, il chercherait où ?

## Installation en une ligne

```bash
bash <(curl -sL https://raw.githubusercontent.com/DimitriGilbert/LimitAID/main/utils/get_limitaid) --install
```

Clone le repo, rend ça exécutable, ajoute le shell completion dans votre `~/.bashrc`. C'est plié. Ou à la mano :

```bash
git clone https://github.com/DimitriGilbert/LimitAID.git
cd LimitAID
./limitaid all
```

## Prérequis

Trois trucs. C'est tout.

* `bash` (4.3+ pour les tableaux associatifs)
* `curl` (appels API)
* `jq` (parsing JSON)

Pas de Node.js. Pas de Python. Pas de npm install. Pas de prise de tête avec un gestionnaire de paquets. Juste bash et deux utilitaires qui sont probablement déjà sur votre machine.

## Pourquoi bash ?

Je sais, je sais. "Mais TypeScript c'est tellement mieux !" "Mais Go ça compile en un seul binaire !" "Mais Rust c'est rapide !"

Oui. Et j'écris dans tous ces langages aussi. Mais pour un truc que je veux installer en 2 secondes, lancer partout, et sans avoir à me prendre la tête avec les versions de runtime ou les cibles de compilation ? Bash, ça hit différemment.

En plus, [parseArger](/blog/Projects/parseArger) me génère tout le parsing d'arguments et le completion, donc la plomberie du CLI était essentiellement gratuite. La logique spécifique à chaque fournisseur, c'est peut-être 100 lignes par service. Le reste, c'est juste du formatage pour faire joli dans un terminal.

Parfois l'outil simple, c'est le bon outil.

## Le nom

LimitAID. Comme "first aid" mais pour vos limites IA. Parce que quand vous êtes dans le flow et que bam, rate limit en pleine figure, ça fait vraiment effet d'urgence médicale. En tout cas pour moi ^^'

## Structure du projet

```text
LimitAID/
├── limitaid          Point d'entrée principal
├── bin/codex         Codex/OpenAI + Spark
├── bin/openrouter    Solde OpenRouter
├── bin/zai           Plan coding Z.ai
├── bin/all           Tous les fournisseurs
├── lib/config.sh     Chargement des clés + auto-découverte
├── lib/http.sh       Helpers HTTP
├── lib/time.sh       Formatage du temps
├── lib/output.sh     Helpers de sortie toute belle
├── completely.bash   Completion bash
└── utils/            Scripts d'installation
```

Propre, petit, facile à piger. Chaque fournisseur a son propre script dans `bin/`, la logique partagée vit dans `lib/`. Le point d'entrée `limitaid` se contente de router vers la bonne sous-commande.

## Sans filtre

LimitAID existe parce que j'en avais ras le bol de prendre des rate limits en plein milieu d'une tâche sans savoir quand ça allait se réinitialiser. C'est un outil né de l'agacement, ce qui, honnêtement, est la façon dont naissent la plupart de mes meilleurs outils.

En tant que quelqu'un qui compte sur l'IA pour bosser à cause de ma SEP, savoir exactement combien de temps de coding il me reste avant qu'une fenêtre se réinitialise — c'est pas un truc sympa à avoir, c'est la différence entre finir une feature aujourd'hui ou perdre une journée entière à attendre. Chaque minute compte quand le brouillard cérébral peut vous tomber dessus à n'importe quel moment.
