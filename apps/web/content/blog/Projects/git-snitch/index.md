Remember [git-moar](/blog/Projects/git-moar)? That little tool I built because staring at `git log` was making my eyes bleed? Well... I got carried away. A *lot* carried away ^^'

What started as "let me slap some charts on git data" became a full-on monorepo with a React renderer, a TanStack Start marketing site, custom TSX templates, AI-powered worklogs, and enough quality metrics to make your project manager cry tears of joy (or horror, depending on your commit history).

Meet **git-snitch**. The thing git-moar grew up to become.

## What changed?

So git-moar was fine! It worked! People (ok, like 3 people, but still!) used it!

But the codebase was... well, it was me learning as I went with AI, which means it was held together by hopes, dreams, and a concerning amount of `// TODO: refactor later` comments that later never came.

git-snitch is the same idea — turn your git history into something readable — but rebuilt from the ground up as a proper monorepo:

* **`@git-snitch/core`** — all the git data crunching, analysis, anonymization
* **`@git-snitch/renderer`** — React + Recharts + TanStack Router, renders standalone HTML
* **`@git-snitch/ui`** — shared shadcn/ui components
* **`@git-snitch/cli`** — the command line interface

Yes, I went full monorepo for a tool that generates a single HTML file. Yes, I see the irony. No, I don't regret it (much).

## Three commands, that's it

```bash
# Analyze a single repository
git-snitch repo

# Scan a directory and discover all git repos
git-snitch scan ~/projects

# Generate an AI worklog from exported data
git-snitch worklog report.json
```

That's the whole CLI. `repo` for one project, `scan` for many, `worklog` when you need a narrative summary instead of charts. Simple.

## The good stuff

### Quality metrics that actually matter

git-snitch calculates real quality signals from your commit history:

* **Bus Factor** — how many people need to disappear before your project dies (spoiler: for most side projects, it's 1, and that 1 is you)
* **Code Churn** — lines added vs deleted over time, with per-contributor breakdowns
* **Health Score** — a single number to tell you if your repo is thriving or on life support
* **Gini Coefficient** — yeah, the economics thing, applied to contribution inequality. Because apparently I needed to make git analytics *academic*

### Privacy-first anonymization

This one I'm actually proud of. Need to share a report but don't want the whole internet knowing your file structure and team emails?

```bash
git-snitch repo --anon
```

One flag. Names become pseudonyms, emails get replaced, file paths get hashed with FNV-1a, commit hashes get scrambled, messages get replaced with classification labels. You can even use `--obfuscate-key` for deterministic hashing so anonymized reports stay comparable across runs.

Or cherry-pick what you want to hide:

```bash
git-snitch repo --hide-emails --hide-urls --hide-messages
```

### AI-powered worklogs

This was the "oh, I can do that?" feature. Feed git-snitch data to an AI harness and get back a narrative work log:

```bash
git-snitch repo --wl-harness opencode --wl-skill work-log
```

Supports [opencode](https://opencode.ai), [pi](https://github.com/nicepkg/pi), and [Codex](https://github.com/openai/codex) as harnesses. Four skill templates: `repo-log`, `work-log`, `changelog`, `devlog`. Pick your flavor.

### Custom templates

Don't like my beautiful (read: functional-ish) default report? Pass a TSX file:

```bash
git-snitch repo --template ./my-template.tsx
```

Missing routes fall back to the built-in renderer, so you can override just the parts you care about without reimplementing everything.

### Multi-repo scanning

The `scan` command recursively discovers git repositories and produces a comparative report. Want to see your entire workspace at a glance?

```bash
git-snitch scan ~/projects --period 3m --max-depth 3
```

Cross-project contributor analysis, comparative activity trends, the works.

## The tech stack

Look, I know what you're thinking. "It's a tool that generates HTML, why does it need all this?" And you're right! But I also wanted to learn things, and the combination of AI making me faster and MS making me slower meant I could afford to go a bit overboard.

* **TypeScript** everywhere, strict mode, no `any` allowed
* **pnpm + Turborepo** monorepo
* **React 19 + Recharts** for the report renderer
* **TanStack Router** for report navigation
* **TanStack Start** for the marketing site
* **shadcn/ui** shared through a workspace package
* **Vite** for building the standalone renderer
* **Commander** for the CLI
* **Zod** for validation
* **Vitest** for testing

Every report is a **single self-contained HTML file**. No server. No database. No hosting required. Open it from your filesystem, email it, stick it on a shared drive. The charts are embedded, the styles are inlined. It just works.

## GitHub enrichment

If `gh` CLI is installed and you have a GitHub remote, git-snitch automatically pulls stars, forks, license, and topics into the report. Don't want that? `--no-github` skips it. Simple.

## Configuration

Create `.git-snitch/config.json` in your repo or scan directory:

```json
{
  "report": {
    "outputPath": "./reports/output.html",
    "theme": "dark"
  },
  "repo": {
    "branches": ["main", "develop"]
  },
  "anon": {
    "hideEmails": true,
    "hideUrls": true
  }
}
```

CLI flags override config values. Sensible defaults everywhere. No config file required to get started.

## What's in a name?

"git-snitch" because it snitches on your git habits. Who commits at 3 AM? Who has 2000-line merge commits? Who hasn't touched a file in 6 months? git-snitch will tell you. It will tell everyone.

(The anonymization feature exists precisely because of this ^^')

