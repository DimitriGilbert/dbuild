You know what I'm tired of? Algorithms deciding what I should watch. I subscribe to a creator because I like their stuff, and then some black box decides that *actually*, what I really want is 47 videos about underwater basket weaving because I watched one (1) video about crafts that one time.

No. I subscribed to people. Show me their stuff. In order. Like a normal feed. Is that so hard?

Well, apparently it is for the major platforms! So I built **[FeedElity](https://github.com/DimitriGilbert/FeedElity)** — a local-first, video-oriented RSS client that follows your creators across YouTube, Odysee, and PeerTube from one place. No algorithm. No ads. No "we think you might also enjoy..." Just your subscriptions, your way.

## What is FeedElity?

At its core, it's a three-column interface:

* **Left**: Your sources/creators
* **Middle**: Content list (newest first, like nature intended)
* **Right**: Viewer/player

You browse the full catalog without an account. Sign in to subscribe, mark favorites, track what you've watched, build playlists, and refresh on *your* terms. Not when Google decides to push notifications.

And here's the kicker — it runs as a web app OR a desktop app. The desktop shell bundles its own backend and SQLite database. Local-first by default. Your data stays on your machine. Point it at a remote server if you want, but you don't have to.

## Why I built this

I had an old version running on a low-code backend that was... fine? It worked. But it had stubs everywhere, features that were half-implemented, and the whole thing was getting harder to maintain than it should've been.

So I did what every sane developer does. I rewrote the whole thing from scratch. (I know, I know. But it was the right call this time, I swear!)

The rewrite goals were simple:
- Match current functionality without losing anything
- Support YouTube, Odysee, and PeerTube from day one
- Make it extensible so adding new sources later is painless
- Keep the three-column, high-density workflow
- Modernize the UI without making it sparse

## The source adapter pattern

This is probably the part I'm most happy with. Each platform (YouTube, Odysee, PeerTube) has its own adapter that implements a common interface:

```typescript
interface SourceAdapter<TSourceType extends SourceType> {
  readonly sourceType: TSourceType;
  detect(input: string): SourceDetectionResult;
  resolveInput(input: DetectedSourceInput): Promise<SourceAdapterResult<ResolvedSourceInput>>;
  normalizeCatalogPayload(input, payload): SourceAdapterResult<NormalizedCatalogPayload>;
  fetchCatalog(input): Promise<SourceAdapterResult<NormalizedCatalogPayload>>;
}
```

Each adapter handles:
- URL detection (is this a YouTube URL? Odysee? PeerTube?)
- Metadata fetching
- Content normalization into a uniform shape
- Playback URL generation

The rest of the app doesn't care where the content comes from. Want to add Rumble? Write an adapter, register it, done. No touching the UI, the database, or anything else.

## The data model

I split things into two layers:

**Global catalog** — Creators, feeds, content items, and playable sources stored in source-neutral tables. Anyone can browse this data without an account.

**User overlays** — Subscriptions, favorites, watched history, playlists, and settings live in separate tables tied to a `userId`. Your data is yours. The API enforces this at the boundary, not just in the UI.

The schema uses Drizzle ORM on SQLite (or libSQL for the server version):

```typescript
export const creator = sqliteTable("creator", {
  id: text("id").primaryKey(),
  sourceType: text("source_type", { enum: sourceTypeValues }).notNull(),
  sourceExternalId: text("source_external_id").notNull(),
  displayName: text("display_name").notNull(),
  description: text("description"),
  imageUrl: text("image_url"),
  // ...
});

export const contentItem = sqliteTable("content_item", {
  id: text("id").primaryKey(),
  creatorId: text("creator_id").notNull().references(() => creator.id),
  title: text("title").notNull(),
  contentType: text("content_type", { enum: contentTypeValues }).default("video"),
  // ...
});
```

Clean, normalized, indexed properly. The refresh system even tracks run history with detailed per-feed results — how many items discovered, created, updated, skipped, succeeded, failed. Full audit trail.

## The stack

I went a different direction from my usual Next.js habit on this one:

| Layer | Choice |
|---|---|
| Runtime | [Bun](https://bun.sh) |
| Frontend | [SolidJS](https://solidjs.com) + Tailwind CSS + TanStack Router/Query |
| Backend | [Hono](https://hono.dev) on Bun |
| API | [oRPC](https://orpc.unnoq.com) — end-to-end type-safe, OpenAPI-compatible |
| Auth | [better-auth](https://better-auth.com) — email/password, self-service sign-up |
| Database | SQLite / libSQL + Drizzle ORM |
| Desktop | [Electrobun](https://electrobun.dev) |
| Monorepo | Turborepo |

Yes, SolidJS. Not React. I know, I surprised myself too. But for a high-density, reactive UI like this, Solid's fine-grained reactivity just feels right. No virtual DOM overhead, updates are surgical. The three-column layout with live updates feels snappy.

And oRPC instead of tRPC? I wanted something OpenAPI-compatible while still being type-safe end-to-end. oRPC gives you both. Plus it plays nice with Hono.

## Runtime modes

This part took some thinking. The app can run in four different modes:

| Mode | Backend | Database |
|---|---|---|
| `local` (browser dev) | Local server at localhost:3002 | Shared `local.db` |
| `web` (deployed) | Remote server | Remote DB |
| `desktop-local` (default) | Embedded Bun/Hono | Local SQLite |
| `desktop-remote` | Configured remote server | Remote DB |

The desktop app bundles its own Bun runtime and Hono server. No external dependencies. It just works. Unless you want it to talk to a shared server, in which case you set one environment variable and off you go.

## Supported sources (for now)

- **YouTube** — RSS channel feeds, iframe embed playback (using youtube-nocookie.com because privacy matters)
- **Odysee** — RSS feeds, native media playback
- **PeerTube** — Instance API for discovery and metadata, embed playback

Each adapter normalizes the platform-specific chaos into a clean, uniform data shape. The YouTube adapter alone is ~430 lines because YouTube's RSS is... well, it's YouTube. You know how it is.

## Ingestion (the no-scheduler approach)

I made a deliberate choice here: no background scheduler. No automatic refreshing. You refresh when *you* want to. Manual refresh respects cadence metadata (some feeds tell you how often to check). Force refresh ignores it.

This is by design. I didn't want FeedElity phoning home on a schedule, especially in desktop-local mode. You own the timing. You own the data.

## Quick start

```bash
bun install
bun run db:local    # start the local SQLite database
bun run db:push     # apply the schema
bun run dev
```

Web app at `http://localhost:3001`, API server at `http://localhost:3002`.

Or run individual targets:

```bash
bun run dev:web        # frontend only
bun run dev:server     # backend only
bun run dev:desktop    # desktop app with HMR
```

Desktop builds for Linux, Mac, and Windows are all supported.

## The honest truth

FeedElity is a rewrite of an existing tool, and those are always tricky. You have to fight the urge to redesign everything while still making the improvements you know are needed. I think I found the right balance — the three-column workflow is the same, but everything under the hood is cleaner, faster, and more maintainable.

But for following video creators across platforms without an algorithm deciding what's "relevant" for you? It does the job. And your data stays yours.
