## C'est quoi FeedElity ?

Au cœur du truc, c'est une interface à trois colonnes :

* **Gauche** : Tes sources/créateurs
* **Milieu** : La liste de contenu (du plus récent en premier, comme la nature l'a prévu)
* **Droite** : Le lecteur/player

Tu peux naviguer le catalogue complet sans compte. Tu te connectes pour t'abonner, marquer tes favoris, tracker ce que t'as regardé, construire des playlists, et rafraîchir quand *tu* le décides. Pas quand Google décide de te push des notifications.

Et le truc qui tue — ça tourne en web app OU en app desktop. Le shell desktop embarque son propre backend et sa base SQLite. Local-first par défaut. Tes données restent sur ta machine. Pointe-le vers un serveur distant si tu veux, mais t'es pas obligé.

## Le pattern d'adaptateur de source

C'est probablement la partie dont je suis le plus content. Chaque plateforme (YouTube, Odysee, PeerTube) a son propre adaptateur qui implémente une interface commune :

```typescript
interface SourceAdapter<TSourceType extends SourceType> {
  readonly sourceType: TSourceType;
  detect(input: string): SourceDetectionResult;
  resolveInput(input: DetectedSourceInput): Promise<SourceAdapterResult<ResolvedSourceInput>>;
  normalizeCatalogPayload(input, payload): SourceAdapterResult<NormalizedCatalogPayload>;
  fetchCatalog(input): Promise<SourceAdapterResult<NormalizedCatalogPayload>>;
}
```

Chaque adaptateur gère :
- La détection d'URL (c'est une URL YouTube ? Odysee ? PeerTube ?)
- La récupération de métadonnées
- La normalisation du contenu en un format uniforme
- La génération des URLs de lecture

Le reste de l'app se fiche de savoir d'où vient le contenu. Tu veux rajouter Rumble ? Écris un adaptateur, enregistre-le, et c'est plié. Pas besoin de toucher l'UI, la base de données, ou quoi que ce soit d'autre.

## Le modèle de données

J'ai séparé les choses en deux couches :

**Catalogue global** — Créateurs, feeds, éléments de contenu et sources jouables stockés dans des tables neutres vis-à-vis des sources. N'importe qui peut naviguer ces données sans compte.

**Overlays utilisateur** — Abonnements, favoris, historique de visionnage, playlists et paramètres vivent dans des tables séparées liées à un `userId`. Tes données sont à toi. L'API le garantit à la frontière, pas juste dans l'UI.

Le schéma utilise Drizzle ORM sur SQLite (ou libSQL pour la version serveur) :

```typescript
export const creator = sqliteTable("creator", {
  id: text("id").primaryKey(),
  sourceType: text("source_type", { enum: sourceTypeValues }).notNull(),
  sourceExternalId: text("source_external_id").notNull(),
  displayName: text("display_name").notNull(),
  description: text("description"),
  imageUrl: text("image_url"),
  // ...
});

export const contentItem = sqliteTable("content_item", {
  id: text("id").primaryKey(),
  creatorId: text("creator_id").notNull().references(() => creator.id),
  title: text("title").notNull(),
  contentType: text("content_type", { enum: contentTypeValues }).default("video"),
  // ...
});
```

Clean, normalisé, indexé comme il faut. Le système de rafraîchissement tracke même l'historique des exécutions avec des résultats détaillés par feed — combien d'éléments découverts, créés, mis à jour, ignorés, réussis, échoués. Audit trail complet.

## La stack

Je suis parti dans une direction différente de mon habitude Next.js sur ce coup-là :

| Couche | Choix |
|---|---|
| Runtime | [Bun](https://bun.sh) |
| Frontend | [SolidJS](https://solidjs.com) + Tailwind CSS + TanStack Router/Query |
| Backend | [Hono](https://hono.dev) sur Bun |
| API | [oRPC](https://orpc.unnoq.com) — type-safe de bout en bout, compatible OpenAPI |
| Auth | [better-auth](https://better-auth.com) — email/password, inscription en libre-service |
| Base de données | SQLite / libSQL + Drizzle ORM |
| Desktop | [Electrobun](https://electrobun.dev) |
| Monorepo | Turborepo |

Oui, SolidJS. Pas React. Je sais, je me suis surpris moi aussi. Mais pour une UI réactive à haute densité comme ça, la réactivité à grain fin de Solid, ça fait du bien. Pas d'overhead de virtual DOM, les mises à jour sont chirurgicales. Le layout à trois colonnes avec les updates en direct, c'est réactif.

Et oRPC au lieu de tRPC ? Je voulais un truc compatible OpenAPI tout en restant type-safe de bout en bout. oRPC te file les deux. En plus il s'entend bien avec Hono.

## Les modes d'exécution

Cette partie m'a demandé un peu de réflexion. L'app peut tourner dans quatre modes différents :

| Mode | Backend | Base de données |
|---|---|---|
| `local` (dev navigateur) | Serveur local sur localhost:3002 | `local.db` partagé |
| `web` (déployé) | Serveur distant | DB distante |
| `desktop-local` (par défaut) | Bun/Hono embarqué | SQLite locale |
| `desktop-remote` | Serveur distant configuré | DB distante |

L'app desktop embarque son propre runtime Bun et son serveur Hono. Pas de dépendances externes. Ça marche, un point c'est tout. À moins que tu veuilles qu'elle discute avec un serveur partagé, auquel cas tu mets une variable d'environnement et c'est parti.

## Les sources supportées (pour l'instant)

- **YouTube** — Feeds RSS de chaînes, lecture en iframe embed (en utilisant youtube-nocookie.com parce que la vie privée c'est pas déconnant)
- **Odysee** — Feeds RSS, lecture média native
- **PeerTube** — API d'instance pour la découverte et les métadonnées, lecture en embed

Chaque adaptateur normalise le chaos spécifique à chaque plateforme en un format de données propre et uniforme. L'adaptateur YouTube à lui tout seul fait ~430 lignes parce que le RSS de YouTube c'est... bah, c'est YouTube. Tu sais comment c'est.

## L'ingestion (l'approche sans scheduler)

J'ai fait un choix délibéré ici : pas de scheduler en arrière-plan. Pas de rafraîchissement automatique. Tu rafraîchis quand *tu* le veux. Le rafraîchissement manuel respecte les métadonnées de cadence (certains feeds te disent à quelle fréquence checker). Le rafraîchissement forcé les ignore.

C'est fait exprès. Je voulais pas que FeedElity téléphone à la maison selon un calendrier, surtout en mode desktop-local. C'est toi qui choisis le timing. C'est toi qui possède les données.

## Démarrage rapide

```bash
bun install
bun run db:local    # démarre la base SQLite locale
bun run db:push     # applique le schéma
bun run dev
```

Web app sur `http://localhost:3001`, serveur API sur `http://localhost:3002`.

Ou lance les cibles individuelles :

```bash
bun run dev:web        # frontend uniquement
bun run dev:server     # backend uniquement
bun run dev:desktop    # app desktop avec HMR
```

Les builds desktop pour Linux, Mac et Windows sont tous supportés.
